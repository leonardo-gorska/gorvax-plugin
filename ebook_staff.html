SEÇÃO 01

Painel Admin — Comandos Exclusivos

Comandos disponíveis exclusivamente para a equipe administrativa. Todos requerem permissão gorvax.admin ou op.

Comandos Gerais de Admin

COMANDO DESCRIÇÃO PERMISSÃO
/gorvax reload Recarrega todas as configurações (config, mensagens, bosses, itens, crates, quests, battlepass, cosméticos, conquistas) gorvax.admin
/gorvax debug Ativa/desativa modo debug com logging detalhado gorvax.admin
/gorvax version Exibe versão do plugin e dependências carregadas gorvax.admin

Comandos de Boss

COMANDO DESCRIÇÃO
/boss spawn <nome> Invoca um World Boss na posição do admin. Nomes: rei_gorvax, indrax, zarith, kaldur, skulkor, xylos, vulgathor
/boss reload Recarrega boss_settings.yml, boss_rewards.yml e limpa cache de loots
/boss testloot <boss> <rank> Testa a loot table de um boss para um rank específico (1–10) sem precisar matá-lo
/boss kill Remove o boss ativo à força (caso esteja bugado)

Comandos de Economia

COMANDO DESCRIÇÃO
/eco give <player> <valor> Adiciona dinheiro à conta do jogador (Vault)
/eco take <player> <valor> Remove dinheiro da conta do jogador
/eco set <player> <valor> Define o saldo exato de um jogador

Comandos de Reino (Admin)

COMANDO DESCRIÇÃO
/reino admin delete <reino> Remove um reino forçadamente (inclui claims e dados)
/reino admin setking <reino> <player> Transfere liderança de um reino
/reino admin addblocks <player> <qtd> Adiciona blocos de claim a um jogador

Comandos de Crate

COMANDO DESCRIÇÃO
/crate give <player> <tipo> <qtd> Dá chaves de crate ao jogador. Tipos: comum, raro, lendario, sazonal
/crate giveall <tipo> <qtd> Dá chaves para todos os jogadores online

Comandos de Battle Pass

COMANDO DESCRIÇÃO
/bp addxp <player> <amount> Adiciona XP do Battle Pass ao jogador
/bp setpremium <player> Ativa o track Premium para o jogador
/bp reset <player> Reseta progresso do Battle Pass do jogador

Outros Comandos Admin

COMANDO DESCRIÇÃO
/endresetdragao Força o reset do End imediatamente (mata dragão, limpa chunks)
/bounty set <player> <valor> Define uma recompensa pela cabeça de um jogador
/duel force <p1> <p2> Força um duelo entre dois jogadores
⚠️ ATENÇÃO
Após usar /gorvax reload, o plugin exibirá: "§a[Gorvax] Configurações recarregadas com sucesso!". Se houver erros no YAML, o console mostrará o arquivo e a linha com problema. Sempre verifique o console após reload.

SEÇÃO 02

Configuração Principal (config.yml)

O arquivo config.yml é o coração do GorvaxCore. Contém todas as configurações gerais do servidor, divididas nas seguintes seções:

Storage (Armazenamento)

storage:
  type: yaml # Opções: yaml, sqlite, mysql
  sqlite:
    file: gorvax.db
  mysql:
    host: localhost
    port: 3306
    database: gorvax
    pool_size: 10

Economia & Reinos

kingdoms:
  creation_cost: 5000.0 # Custo para criar um reino
  upkeep_enabled: true
  upkeep_per_chunk: 50.0
  tax_rate: 0.05 # 5% de taxa do banco
  max_members: 30
  max_claims: 100
  visit_cost: 500.0
  visit_cooldown: 60 # segundos

Chat & Comunicação

chat:
  local_radius: 200 # Alcance do canal local em blocos
  format: "{kingdom_tag} {rank} {player}: {message}"
  rank_icons:
    king: "👑"
    vice: "⚔️"
    noble: "🛡️"
    resident: "🏠"
  anti_spam:
    cooldown_ms: 1500
    max_caps_percent: 60
    block_links: true

Combate

combat:
  tag_duration: 15 # segundos de combat tag
  logger_enabled: true # NPC ao deslogar em combate
  logger_duration: 15
  killstreak:
    enabled: true
    rewards:
      3: 150.0 # 3 kills = $150
      5: 400.0
      10: 1000.0

VIP & Ranks Premium

vip:
  enabled: true
  tiers:
    vip:
      display_name: "§a[✦ VIP]"
      extra_claim_blocks: 500
      extra_homes: 2
    vip_plus:
      extra_claim_blocks: 1500
      extra_homes: 4
      market_discount_percent: 5
    elite:
      extra_claim_blocks: 3000
      extra_homes: 6
      market_discount_percent: 10
    lendario:
      extra_claim_blocks: 5000
      extra_homes: 10
      market_discount_percent: 15

Outras Configurações Importantes

  • Boss Events: boss_events.schedule_enabled: true — Invocação automática via cron
  • Scoreboard: scoreboard.enabled: true, update_interval: 20 ticks
  • Duelos: duel.expiration: 60s, bet_enabled: true, max_bet: 10000.0
  • Tutorial: tutorial.enabled: true — Sistema de onboarding com welcome kit
  • Daily Rewards: daily_rewards.enabled: true — 7 dias de recompensas progressivas
  • Battle Pass: battle_pass.enabled: true — Temporadas de 30 dias, 30 níveis
  • Leaderboards: leaderboards.enabled: true, update_interval: 300s
  • Discord: discord.enabled: false — Webhook para chat e boss kills

SEÇÃO 03

Arquivos de Configuração de Conteúdo

Além do config.yml, o GorvaxCore utiliza diversos arquivos YAML para definir conteúdo do jogo. Todos ficam em plugins/GorvaxCore/.

📄 boss_settings.yml
Configurações de stats, skills e efeitos visuais dos 7 World Bosses: Rei Gorvax (1200 HP), Indrax (1100 HP), Zar'ith (800 HP), Kaldur (900 HP), Skulkor (1000 HP), Xylos (950 HP) e Vulgathor (1050 HP). Cada boss possui habilidades únicas com dano, raio, duração e cooldown configuráveis, além de efeitos visuais como partículas, auras e eventos atmosféricos.
📄 boss_rewards.yml
Loot tables completas por boss e rank. Cada boss oferece: armas lendárias (espada, arco, machado), set lendário (capacete, peitoral, calça, botas, escudo), ferramentas lendárias (picareta, machado) e itens comuns. MVP (Top 1) recebe 1 arma lendária aleatória. Top 1–3 recebem peças do set. Todos recebem itens comuns baseados no rank de dano.
📄 custom_items.yml
Define 5 itens lendários: Lâmina de Gorvax (NETHERITE_SWORD, Sharpness 7, Fire Aspect 3, Wither on-hit 30%), Coroa de Indrax (NETHERITE_HELMET, +4 HP, Water Breathing), Botas Velocistas (NETHERITE_BOOTS, Speed 1 permanente), Arco do Caçador (BOW, Power 6, Infinity, Arrow Trail), Escudo do Guardião (SHIELD, +20% knockback resist). Cada item tem base, enchants, atributos, efeitos e source.
📄 crates.yml
4 tipos de crates com reward tables: Comum (dinheiro $100–$250, blocos 50–100, itens básicos), Raro (dinheiro $500–$1000, blocos 200–500, Netherite, Elytra 2%), Lendário (dinheiro $2500–$5000, custom items 3–4%, Netherite x4, broadcast ativado), Sazonal (dinheiro $1000–$3000, Totem, God Apples, títulos exclusivos). Cada reward tem peso (weight) que define probabilidade.
📄 cosmetics.yml
Cosméticos organizados em 5 categorias: Walk Particles (rastros ao caminhar), Kill Particles (efeitos ao matar), Arrow Trails (trails de flechas), Chat Tags (prefixos no chat), Kill Effects (efeitos na morte da vítima). Cada cosmético tem source: achievement, shop, crate ou vip.
📄 achievements.yml
10 conquistas em 4 categorias: Exploração (Primeiro Passo, Desbravador, Arquiteto), Combate (Caçador de Dragões, Guerreiro, MVP), Economia (Magnata, Primeira Venda), Social (Socialite), Reinos (Rei dos Reis, Veterano). Recompensas incluem blocos, títulos, dinheiro. Tipos de trigger: CLAIM_CREATED, BLOCKS_BROKEN, KILLS, BOSSES_KILLED, MONEY_EARNED, etc.
📄 battlepass.yml
Battle Pass sazonal com 30 níveis, tracks Free e Premium. XP por ação: kill player (50), kill mob (5), mine block (1), boss kill (200), daily claim (30), login (20), quest daily (40), quest weekly (150), duel win (75). XP por nível: base 100 + 20 por nível. Temporada de 30 dias. Recompensas premium incluem cosméticos e títulos exclusivos.
📄 quests.yml
Sistema de quests com 3 diárias (de pool de 20+) e 1 semanal (de pool de 5). Reset diário à meia-noite, semanal na segunda. Tipos: KILL_MOB, MINE_BLOCK, KILL_PLAYER, SELL_MARKET, BOSS_PARTICIPATE, DAILY_COMPLETE. Recompensas: dinheiro ($100–$800), claim blocks, crate keys.
📄 mini_bosses.yml
4 mini-bosses por bioma: Guardião do Deserto (HUSK, 300 HP, deserto/badlands), Sentinela Gélida (STRAY, 400 HP, neve/gelo), Araña da Selva (CAVE_SPIDER, 250 HP, jungle), Fantasma do Nether (BLAZE, 500 HP, nether). Spawn natural a cada 2 min, máx 8 ativos, distância mínima 100 blocos de claims, chance 35%.
📄 market_global.yml
Mercado global com 7 categorias: Blocos (20 itens), Natureza (20 itens), Minérios (15 itens), Alimentos (20 itens), Redstone (18 itens), Decoração (20 itens), Drops/Alquimia (18 itens), Equipamentos (16 itens). Cada item tem preço de compra e venda configuráveis.
📄 messages.yml
Todas as mensagens do plugin (800+ linhas). Categorias: admin, proteção, reinos, plots, trust, confirmação, mercado, bosses, menus, manutenção. Usa códigos de cor Minecraft (§). Suporta placeholders como {0}, {1} para valores dinâmicos.
💡 DICA
Todos os arquivos YAML suportam /gorvax reload para hot-reload sem reiniciar o servidor. Verifique sempre o console após o reload para detectar erros de sintaxe.

SEÇÃO 05

Storage & Infraestrutura

O GorvaxCore suporta 3 modos de armazenamento configuráveis em config.yml → storage.type:

YAML (Padrão)

  • Vantagem: Zero dependências, funciona imediatamente
  • Desvantagem: Mais lento para servidores com 50+ jogadores simultâneos
  • Quando usar: Servidores pequenos e médios (< 50 jogadores)
  • Arquivos: kingdoms.yml, claims.yml, playerdata.yml, market_global.yml, boss_data.yml

SQLite

  • Vantagem: Mais rápido que YAML, arquivo único (gorvax.db)
  • Desvantagem: Não suporta acesso multi-servidor
  • Quando usar: Servidores médios (50–100 jogadores)

MySQL

  • Vantagem: Suporta multi-servidor (BungeeCord/Velocity), melhor performance
  • Desvantagem: Requer servidor MySQL externo
  • Quando usar: Redes de servidores, servers com 100+ jogadores
  • Pool Size: Recomendado 10 para maioria dos servidores
# Configurar MySQL:
storage:
  type: mysql
  mysql:
    host: localhost
    port: 3306
    database: gorvax
    username: gorvax_user
    password: sua_senha_segura
    pool_size: 10
⚠️ MIGRAÇÃO
Ao trocar de storage type, é necessário migrar manualmente os dados. Não há ferramenta de migração automática — planeje com antecedência e faça backup antes de alterar.

Padrão de I/O Assíncrono

O GorvaxCore usa um padrão de Main Thread snapshot → Async I/O:

  • Dados são lidos na thread principal (snapshot)
  • Escrita (save) acontece de forma assíncrona para não bloquear o tick
  • Cache em memória usa ConcurrentHashMap para thread-safety
  • Claims usam cache por chunk para lookup O(1)

SEÇÃO 06

Arquivos de Dados Runtime

Estes arquivos são gerados automaticamente pelo plugin. Não edite manualmente com o servidor online.

💾 kingdoms.yml
Dados de todos os reinos: nome, líder, membros com ranks, banco, claims, diplomacia, PvP interno, nível e histórico de upkeep.
💾 claims.yml
Mapa de claims por chunk. Cada entry: reino dono, subplots, permissões de trust (BUILD, INTERACT, CONTAINER, ACCESS), flags e data de criação.
💾 playerdata.yml
Dados de cada jogador: reino, blocos de claim, homes, títulos, cosméticos ativos, conquistas, estatísticas, battle pass, quests e daily rewards.
💾 boss_data.yml
Estado dos bosses: último spawn, cooldowns, eventos atmosféricos, estatísticas globais (mortes por boss, participações).
⚠️ BACKUP
Faça backup diário! Em caso de corrupção, dados de jogadores, reinos e claims podem ser perdidos. Recomendação: backup a cada 6h com rotação de 7 dias.
⚔

SEÇÃO 09

Proteção Anti-Exploit

Proteções integradas contra griefing e exploits em config.yml → protection:

protection:
  block_wither_spawn: true # Impede Wither em claims
  block_crystal_placement: true # Bloqueia End Crystal
  block_tnt_from_outside: true # Rastreia TNT de fora
  block_bed_explosions: true # Protege camas/âncoras

Trust por Claim

  • BUILD: Colocar e quebrar blocos
  • INTERACT: Portas, botões, alavancas
  • CONTAINER: Baús, fornos, dispensers
  • ACCESS: Entrar no claim (se restrito)

Anti-Cheat Integrado

  • Combat Logger: NPC ao deslogar em combate (15s)
  • Anti-Spam: Cooldown 1.5s, bloqueio caps 60%+, bloqueio links
  • Validação de Saldo: Verifica antes de transações
  • Overlap Check: Impede claims sobrepostos
  • Input Validation: Sanitização contra injection
ℹ️ CACHE
Sistema usa cache por chunks para lookup O(1): Chunk Hash → Claim → Subplot → Trust Check

SEÇÃO 11

Reset do End

O GorvaxCore inclui reset automático do End para renovar recursos e gerar novos desafios.

end_reset:
  enabled: true
  interval_days: 7 # A cada 7 dias
  warning_minutes: 30 # Aviso 30 min antes
  teleport_players_out: true # Teleporta jogadores ao spawn
  regenerate_dragon: true

Processo de Reset

  1. Broadcast de aviso 30 min antes: "§4§l⚠ O End será resetado em 30 minutos!"
  2. Avisos periódicos (15 min, 5 min, 1 min)
  3. Teleporte forçado de todos os jogadores no End
  4. Deleção e regeneração dos chunks do End
  5. Re-spawn do Ender Dragon
  6. Broadcast: "§a§l✔ O End foi resetado com sucesso!"

Comando Manual

COMANDO DESCRIÇÃO
/endresetdragao Força o reset imediatamente (requer gorvax.admin)
⚠️ CUIDADO
O reset apaga TODOS os blocos no End (incluindo construções de jogadores). Avise com antecedência e considere desativar claims no End.

SEÇÃO 12

Audit Logs

O GorvaxCore registra eventos administrativos automaticamente no console e em arquivos de log para rastreabilidade.

Eventos Registrados

  • Economia: Todas as transações de /eco give/take/set com jogador, valor e admin
  • Reinos: Criação, deleção forçada, transferência de liderança
  • Claims: Remoção forçada de claims por admin
  • Bosses: Spawns manuais e kills forçados
  • Crates: Keys distribuídas via admin (/crate give/giveall)
  • Battle Pass: XP adicionado, Premium ativado, resets
  • Punições: Bans, mutes e warnings com motivo

Formato de Log

# Exemplo de log entry:
[2024-01-15 14:32:18] [ADMIN] StaffPlayer: /eco give Jogador1 5000.0
[2024-01-15 14:35:42] [BOSS] StaffPlayer: /boss spawn rei_gorvax (x:100, z:-200)
[2024-01-15 15:01:03] [CRATE] StaffPlayer: /crate give Jogador2 lendario 3
💡 DICA
Os logs ficam em plugins/GorvaxCore/logs/ com rotação diária. Mantenha pelo menos 30 dias de histórico para auditorias.

SEÇÃO 13

Compatibilidade Bedrock

O GorvaxCore é compatível com jogadores Bedrock via Geyser + Floodgate.

Funcionamento

  • AnvilGUI Fallback: Menus que usam AnvilGUI (renomear itens, inserir valores) automaticamente abrem em modo chat para jogadores Bedrock, pois AnvilGUI não é compatível com Bedrock Edition
  • Detecção Automática: O plugin detecta jogadores Bedrock via Floodgate API (FloodgateApi.getInstance().isFloodgatePlayer(uuid))
  • Prefixo: Jogadores Bedrock são identificados pelo prefixo . (ponto) no username

Requisitos

PLUGIN VERSÃO FUNÇÃO
Geyser 2.0+ Traduz protocolo Bedrock → Java
Floodgate 2.0+ Autentica jogadores Bedrock sem conta Java

Limitações Conhecidas

  • Partículas de cosméticos podem não renderizar corretamente no Bedrock
  • Alguns emojis de rank no chat podem não aparecer
  • Menus GUI podem ter layout diferente
ℹ️ SOFT DEPENDENCY
Floodgate é uma dependência opcional. Se não estiver instalado, o fallback é desativado e jogadores Bedrock sem AnvilGUI receberão mensagem de erro.

Troubleshooting & Conflitos

Problemas Comuns

❌ Plugin não carrega

  • Verifique se Java 21+ está instalado: java -version
  • Verifique se Paper 1.21+ é o server JAR (não Spigot/Bukkit)
  • Verifique se Vault, WorldGuard e PlaceholderAPI estão instalados
  • Confira o console para stack traces de erros

❌ Economia não funciona

  • Vault precisa de um plugin de economia backend (EssentialsX, CMI, etc.)
  • Verifique /vault-info para confirmar que o provedor está ativo

❌ Boss não spawna

  • Verifique se não há outro boss ativo (apenas 1 por vez)
  • Verifique se o mundo não está com doMobSpawning: false
  • Use /boss spawn <nome> para teste manual

❌ Erro de YAML após edição

  • Nunca use TABs — YAML só aceita espaços
  • Strings com caracteres especiais devem estar entre aspas
  • Use um validador YAML online antes de salvar
  • O console mostrará o arquivo e a linha exata do erro

❌ Placeholders não funcionam

  • Verifique se PlaceholderAPI está instalado e ativo
  • Execute /papi ecloud download GorvaxCore se necessário
  • Teste com /papi parse me %gorvax_kingdom%

Conflitos Conhecidos

PLUGIN CONFLITO SOLUÇÃO
GriefPrevention Sistema de claims conflita com o GorvaxCore Desative o GP; o GorvaxCore já tem sistema próprio
Towny Sistema de cidades/nações sobrepõe reinos Não usar ambos — são sistemas mutuamente exclusivos
mcMMO Pode interferir com XP de battle pass Sem conflito grave, mas monitore XP rates
CMI Economy Formato de saldo pode conflitar Configure ambos com a mesma precisão decimal
⚠️ REGRA GERAL
Evite instalar plugins que replicam funcionalidades do GorvaxCore (claims, reinos, economia customizada, partículas). Isso causa conflitos de listeners, dupla execução de eventos e comportamento imprevisível.

Migração Automática de Configs

O GorvaxCore inclui um sistema de migração automática que garante compatibilidade ao atualizar o plugin. Configs antigos são migrados automaticamente, preservando valores customizados.

Como Funciona

  1. Ao iniciar, verifica config_version no config.yml
  2. Se a versão é menor que a mais recente, cria backup automático (config.yml.backup-YYYY-MM-DD)
  3. Aplica migrações sequenciais (V0→V1, V1→V2, etc.) sem sobrescrever valores existentes
  4. Atualiza config_version e salva
  5. Também migra messages.yml com novas mensagens

Backup Automático

  • Backup criado antes de qualquer alteração: config.yml.backup-2026-03-10
  • Se já existe backup do dia, sufixo numérico: config.yml.backup-2026-03-10-1
  • Se o backup falha, a migração é cancelada por segurança

Comando Admin

Comando Descrição Permissão
/gorvax migrateconfig Força re-migração de configurações gorvax.admin

Mensagens no Console

[ConfigMigrator] config.yml está na v0, versão mais recente é v1.
[ConfigMigrator] Backup criado: config.yml.backup-2026-03-10
[ConfigMigrator] Aplicando migração: V0 → V1
[ConfigMigrator] Migração concluída! 1 step(s) aplicado(s).

Extensibilidade

Para adicionar novas migrações em versões futuras:

  1. Criar nova classe em migration/migrations/ (ex: V1_to_V2.java)
  2. Implementar MigrationStep com fromVersion(), toVersion() e apply()
  3. Registrar no ConfigMigrator.registerSteps()
  4. Incrementar LATEST_VERSION no ConfigMigrator

Arquivos

Arquivo Função
MigrationStep.java Interface para cada patch de migração
ConfigMigrator.java Motor principal (backup, sequência, save)
V0_to_V1.java Primeira migração (config_version + chaves B32+)
💡 DICA
Use /gorvax migrateconfig após restaurar um backup antigo para re-aplicar automaticamente todas as migrações necessárias.

Códex de Gorvax — Configuração Admin

O Códex de Gorvax é configurado via codex.yml. Cada entrada é organizada em categorias com tipo de desbloqueio, alvo e quantidade necessária.

Estrutura do codex.yml

categories:
  mobs:
    name: "§c⚔ Bestiário"
    icon: ZOMBIE_HEAD
    entries:
      zombie:
        name: "§fZombi"
        description: "O morto-vivo mais comum..."
        unlock_type: KILL_MOB
        unlock_target: ZOMBIE
        unlock_amount: 10

Tipos de unlock_type

Tipo Disparo unlock_target
KILL_MOB Matar N mobs EntityType (ex: ZOMBIE, ENDERMAN)
MINE_BLOCK Minerar N blocos Material (ex: DIAMOND_ORE)
VISIT_BIOME Visitar bioma Biome ID (ex: DESERT)
BOSS_PARTICIPATE Participar de boss Boss ID ou ANY

Como Adicionar Novas Entradas

  1. Abra codex.yml na pasta do plugin
  2. Adicione uma entrada dentro de uma categoria existente ou crie nova categoria
  3. Defina name, description, unlock_type, unlock_target e unlock_amount
  4. Use /gorvax reload para recarregar

Persistência

Entradas desbloqueadas são salvas no playerdata.yml sob a chave unlockedCodex:

<uuid>:
  unlockedCodex:
    - "mobs.zombie"
    - "mobs.enderman"
    - "biomes.desert"

Arquivos

Arquivo Função
codex.yml Definição de categorias e entradas
CodexManager.java Lógica de unlock, progresso e recompensas
CodexGUI.java Interface gráfica do Códex
CodexCommand.java Comando /codex
CodexListener.java Hooks automáticos (kills, mining, etc.)
